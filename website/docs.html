Quick Start

Becket is built for developers using Cursor or Claude Code. The agent calls MCP tools automatically — you index once, then the agent pulls verified context before every edit.

Example — Cursor

.cursor/mcp.json
{
  "mcpServers": {
    "becket": {
      "command": "becket-mcp",
      "env": {
        "BECKET_ROOT": "${workspaceFolder}"
      }
    }
  }
}

Example — Claude Code / Desktop

~/.claude/claude_desktop_config.json
{
  "mcpServers": {
    "becket": {
      "command": "becket-mcp",
      "env": {
        "BECKET_ROOT": "/path/to/your/repo"
      }
    }
  }
}

Same server for Codex CLI, Windsurf, Cline, Zed, or any host that supports MCP over stdio — point BECKET_ROOT at your repo and run becket build first.

Installation

Becket is free and open source (Apache-2.0). No account, no subscription, no telemetry. You need Node.js/npm for the install wrappers — Rust is only required if you build from source.

Cursor / Claude Code

Install both packages globally so MCP finds them on PATH.

terminal
npm install -g becket becket-mcp

Try without global install

Downloads the native binary on first run. Good for a one-off index or the demo folder.

terminal
npx becket build

Overview

Becket is a local intelligence layer that combines a verified code graph, a graph-grounded knowledge wiki, and context assembly that returns real source snippets for AI agents and developers.

The primary workflow is MCP inside your IDE: the agent calls get_context before modifying code. The CLI is for indexing, inspection, CI gates, and scripts — it exposes the same queries but does not replace an AI coding agent.

MCP Integration

Becket exposes an MCP server over stdio. This is how Cursor and Claude Code agents access verified context automatically — no manual copy-paste.

becket-mcp
Tool Description When to use
get_context Wiki + code snippets + impact, packed to budget Before fixing or refactoring a symbol
get_wiki Grounded markdown page or index router Onboarding; enrich=true fills intent/gotchas
get_impact Change impact analysis Before renaming or deleting a symbol
get_flow Business flow reconstruction Understanding end-to-end paths
get_dependencies Direct and transitive dependencies Dependency exploration

Wiki prose and summaries use MCP sampling — your host agent's model writes intent and gotchas. Becket ships no LLM, holds no API keys, and never generates code — only slices real source from disk.

How does the agent call get_context?

Becket does not hook into every keystroke. MCP exposes tools to Cursor or Claude Code; the agent decides when to call them — same as any other MCP tool. Tool names and descriptions tell the model what is available; it invokes get_context when your task needs codebase context (fix a bug, refactor a symbol, etc.).

For reliable use, add a project rule or ask explicitly — e.g. “use Becket get_context on PaymentService before editing”. Without that, the agent may skip the tool and guess from open files alone.

Cursor — project rule (recommended)

.cursor/rules/becket.mdc
Before modifying code, call the Becket MCP tool get_context
on the relevant symbol (function, class, or module).
Use get_impact before renames or deletions.
Run becket build if .becket/ is missing.

Claude Code: add the same instruction in CLAUDE.md or your project system prompt.

CLI vs MCP: becket context returns the same bundle as get_context. MCP lets the agent fetch it inside the chat; the CLI is for you in the terminal. Neither runs by itself — MCP needs an agent turn; CLI needs you to run the command.

CLI Reference

The CLI indexes your repo and answers queries. Run it from the repository root (or pass --repo).

Index the repository

terminal
becket build
becket build --watch    # incremental rebuild on file changes

Generates versioned JSON artifacts and grounded wiki pages under .becket/. No LLM required.

Context bundle

terminal
becket context capture --task fix
becket context capture --task refactor
becket context capture --task onboard
becket context capture --auto-budget --task fix
becket context capture --plan --task fix
becket context capture --json

Token budget

Task Default budget Focus
fix 6,000 Callers, test snippets, shallow impact
refactor 12,000 Deep impact, more snippets
onboard 8,000 Flow overview + fewer snippets

Query impact

becket impact capture --depth 3

Returns affected modules, downstream symbols, related tests, and risk zones.

Understand a flow

becket flow services

Shows end-to-end execution path and service interactions.

Knowledge wiki

becket wiki sync
becket wiki lint
becket wiki lint --strict
becket wiki show services

Sync recompiles wiki structure from the graph. Lint checks pages against live call edges — see Wiki section below.

Domain refinement (optional)

becket domain rename <auto-id> payment
becket domain add payment src/billing/** capture

Zero-config by default. Refine auto-discovered domains via CLI.

Grounded Repo Wiki

Markdown pages for modules, services, and flows — each anchored to real symbol IDs from the deterministic graph. Structure is compiled from the graph; optional prose (intent and gotchas) via MCP sampling.

Advanced

Incremental rebuild

becket build --watch

Re-parses changed files and auto-syncs stale wiki structure. Run in a terminal while you code.

Multi-repo workspaces

becket workspace build

Links repositories via HTTP, gRPC, or message queues. Requires a becket.workspace.toml manifest.

Keep .becket/ local

Add .becket/ to .gitignore. It is a local cache rebuilt by becket build — not source of truth for the team. The wiki inside is scaffolding for context assembly, not documentation to publish. Re-index after pulls or large refactors; use becket build --watch while coding if you want it always fresh.

Platform support

FAQ & Troubleshooting

Do I need Rust installed?

No. npx becket build and npm install -g becket download prebuilt binaries. Rust is only needed to compile from source.

MCP tools return "index missing"

Run becket build in the repository root first. MCP reads from .becket/ — without a build, there is nothing to query.

becket-mcp command not found

Install globally: npm install -g becket-mcp. Ensure your npm global bin directory is on PATH. Restart Cursor/Claude Code after installing.

Can I use Becket without Cursor or Claude Code?

Yes — the CLI provides context, impact, and flow for terminal workflows. But Becket shines when an AI agent calls MCP tools automatically; the CLI alone does not edit code or run an LLM.

Does Becket send my code to the cloud?

No. All analysis runs locally. Optional wiki prose enrichment uses your MCP host's model (Cursor/Claude) — Becket itself holds no API keys and sends no telemetry.

What symbol name should I use?

Use a function, class, or module name visible in your source — e.g. capture, PaymentService, or a fully qualified name. Run becket build first; symbols come from the indexed graph.

Does get_context run automatically on every edit?

No. Cursor and Claude Code expose Becket as MCP tools; the agent calls them when it decides they are needed (or when your rules/prompt tell it to). Add a project rule or ask explicitly for best results. Becket is not a background daemon inside the IDE.

Should I commit .becket/ or .becket/wiki/?

No, for the typical workflow. Keep .becket/ in .gitignore and run becket build locally (or --watch). Committing wiki pages is only for niche teams that want shared prose in git — not required to save tokens or speed up your agent.

Design Principles

Deterministic First

Core analysis is deterministic. Structure is derived from code, not guessed.

AI-Augmented, Not AI-Dependent

AI enhances interpretation via MCP sampling. The tool works fully without it.

Local-First

All analysis runs locally. Privacy, speed, reproducibility. Zero telemetry.

Machine-Readable Outputs

JSON-compatible, stable schema, versioned artifacts.

Graph-Grounded Wiki

Wiki pages anchor to symbol IDs and are lint-checked against the live graph.

Architecture

Rust-first deterministic core, graph-grounded wiki, context assembly, SQLite index, MCP server. Multi-repo workspaces and cross-service linking.
